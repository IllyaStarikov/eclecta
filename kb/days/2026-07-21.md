### [–end-of-options](https://nesbitt.io/2026/07/21/end-of-options.html) — 8/10

If your tool forks git with an untrusted URL or ref, one of nineteen surveyed package managers actually uses the flag that stops that string being parsed as an option, so the safe pattern is almost universally missing from code you depend on.

Notes: git has two distinct terminators: `--` separates revisions from pathspecs (so `git log main -- README.md` reads as commits on main touching that file), leaving the revision slot with no terminator; `--end-of-options`, added in git 2.24.0 (Nov 2019), fills that gap. The safe invocation for an untrusted revision is `git log --end-of-options "$rev" -- "$path"`, with both markers doing separate jobs. · Subcommand support was staggered: rev-parse only got the flag in 2.30.0 (its own hand-rolled parser), and checkout/reset rejected it until 2.43.1 (Feb 2024) because they parse `--` themselves and left `--end-of-options` in the argv where their parsers choked. · CWE-88 argument injection needs no shell: the wrapper builds a clean argv and calls exec directly, but git parses a dash-leading argument as an option such as `--upload-pack=<cmd>`, `-c core.sshCommand=<cmd>`; hg has `--config=alias...=!<shell>`, ssh has `-oProxyCommand=<cmd>`. CVE-2019-13139 hit `docker build` this way via a git URL `#ref:dir` fragment. · On one day in Aug 2017, git, Mercurial, Subversion and CVS disclosed near-identical ssh `-oProxyCommand` hostname injections (CVE-2017-1000117 / -1000116 / CVE-2017-9800 / -12836); per Phabricator's post-mortem only Subversion fixed it with `--`, git and Mercurial validated hostname format instead because `--` isn't supported by every ssh implementation. · Of 19 package managers checked, 17 fork the git binary by default; only Go's cmd/go uses `--end-of-options`, added across the board in Jan 2026 as the CVE-2025-68119 fix (plus `HGPLAIN=+strictflags` for hg). Cargo defaults to libgit2 and Poetry to dulwich (since 1.2.0). Most other guards (Bundler, CocoaPods, Poetry) were added only as CVE patches, not in original implementations. · Composer's CVE-2022-24828 advisory names `--end-of-options` as the correct fix but rejects dash-leading branch names instead because it supports pre-2.24.0 git; the author's own Homebrew PR raises the floor only to 2.30.0 and leaves checkout/reset alone, since covering them needs 2.43.1, still ahead of several supported distros (Ubuntu 20.04 ships 2.25.1).

### [Meta Garbage Collection: Using OCaml's GC to GC Rust](https://soteria-tools.com/blog/meta-garbage-collection) — 7/10

A verification tool cut a common quadratic blowup to linear in about 40 lines by making OCaml's garbage collector reclaim Tree Borrows state, turning a 3.02s run into 0.54s.

Notes: Soteria Rust is an OCaml symbolic-execution tool that checks Rust programs against Tree Borrows, tracking every reference to a memory location as a tree of five-state nodes; each read/write updates node states, and reaching the fault state signals UB. · The slowdown came from unoptimized MIR: the tool cannot enable compiler optimizations (inlining, dead-code elimination) because they can erase the UB it hunts, so Range::next reborrows persist. Each Iterator::next call adds 9 nodes and ~10 accesses per iteration, giving 9N+10 nodes and 10N+12 accesses (9,010 nodes, 10,012 accesses at N=1000), roughly 45N-squared node accesses. · Soundness argument for dropping nodes: UB is only reachable via a local access, a finished loop iteration's branch has no live references, protected nodes (on function entry) are kept alive by the interpreter, children are always at least as close to UB as parents, and pre-branch state references prevent cross-branch collection in symbolic execution. · Implementation stores the tree as a weak map plus Weak sets and Ephemeron so nodes are held strongly only where their Rust reference is live; a node keeps all ancestors directly, so GCing an intermediate node preserves local/foreign relationships. · OCaml's periodic GC initially ran too rarely (low allocation) and left only the weak-reference cost, so they force Gc.major() when tree size exceeds threshold T, with backoff doubling T to 2T when a run frees nothing; T=256 gave the best balance, 0.54s at N=1000 (5.6x) and 10.6x at N=2000. · On a Tokio init snippet scaled from 64 to 1024, time dropped from 14.8s to 1.7s (8.7x, self-reported); a comparable unnamed tool that does not track Tree Borrows took 15.4s. Some benchmarks got slower from weak-reference overhead. Miri already GCs these trees but must implement it manually since its host language is Rust.
